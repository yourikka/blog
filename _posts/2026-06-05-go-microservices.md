> 微服务不是把项目拆得越碎越好，而是把不同职责的业务能力拆开，让它们能独立开发、独立部署、独立扩容。

我以前看微服务，第一反应是很多很硬的词：注册中心、负载均衡、RPC、网关、熔断、限流、链路追踪、消息队列。看起来很复杂，也很容易让人误以为“把这些组件都用上了，就是微服务”。

后来再看 Martin Fowler、microservices.io、Microsoft 和 AWS 的一些架构资料，我发现微服务真正想解决的不是“技术栈高级不高级”，而是系统变大以后怎么继续维护。一个项目小的时候，所有代码放在一起最方便；但业务越来越多、团队越来越多、发布越来越频繁之后，所有东西绑在一起就会越来越难改。

可以把微服务想成一家餐厅。刚开店时，一个人点餐、做饭、收钱、打包都能完成。但店越做越大，就要分工：前台负责点餐，后厨负责做菜，收银负责付款，配送负责送餐。每个岗位负责自己的事情，协作靠清楚的流程。微服务也是类似的思路：用户服务、订单服务、支付服务、库存服务，各自负责一块业务，通过接口协作。

## 什么是微服务

微服务是一种系统拆分方式。它会把一个大应用拆成多个小服务，每个服务负责一块清楚的业务，并且可以独立运行。

比如一个电商系统，单体项目里可能什么都有：用户、商品、订单、支付、库存、通知。微服务会把它们拆开：

- 用户服务负责用户资料和账号状态。
- 商品服务负责商品信息和价格。
- 订单服务负责下单和订单状态。
- 支付服务负责支付单和支付回调。
- 库存服务负责库存扣减和库存释放。
- 通知服务负责短信、邮件和站内信。

这样拆的重点不是“文件夹变多了”，而是每个服务都有自己的职责。订单服务不要直接改库存表，支付服务也不要随便改订单表。它们要通过接口或消息协作。

所以微服务最核心的一句话是：围绕业务能力拆分，而不是围绕技术层拆分。

不要把系统拆成 Controller 服务、Service 服务、DAO 服务。这种拆法只是把原来的三层架构搬到了网络上，复杂度会更高，收益却很小。

## 什么时候需要微服务

不是所有项目都需要微服务。

如果一个项目还很小，业务还在快速变化，团队也只有几个人，单体通常更舒服。因为单体开发快、调试简单、部署也简单。这个阶段强行上微服务，反而会把时间花在注册中心、配置、链路追踪、部署脚本上。

更适合考虑微服务的情况通常有几种。

第一，业务边界已经比较清楚。比如订单、支付、库存之间的职责比较稳定，不是今天这样拆，明天又完全换一套。

第二，某些模块变化特别频繁。比如活动规则经常改，支付模块却要求稳定。把它们放在一起发布，会让稳定模块经常被打扰。

第三，某些模块压力明显不同。比如商品查询流量很高，报表导出计算很重，通知服务适合异步处理。拆开后可以单独扩容。

第四，团队协作需要边界。不同团队负责不同服务，可以减少互相等待。

第五，已经有一定工程基础。至少要有日志、监控、测试、CI/CD、部署和回滚能力。没有这些能力，微服务会很难排查。

我的理解是：先能把单体写清楚，再考虑拆微服务。如果单体里模块都乱，拆成微服务只会变成“分布式混乱”。

## 微服务的优点

微服务的第一个优点是能独立上线。

单体里改一个小功能，也可能要把整个系统重新发布。系统越大，发布越让人紧张。微服务拆得合理时，通知服务改短信模板，不需要订单服务一起发布。

第二个优点是职责更清楚。

以前所有代码在一个项目里，很容易互相乱调用。订单逻辑可能直接动库存数据，支付逻辑可能直接改订单状态。微服务会强迫我们把边界想清楚：谁拥有数据，谁负责规则，谁对结果负责。

第三个优点是可以单独扩容。

如果商品查询很忙，就扩商品服务；如果通知任务堆积，就扩通知服务；如果支付服务压力不大，就不用跟着一起扩。资源可以用得更精确。

第四个优点是故障影响更容易控制。

推荐服务挂了，可以返回默认推荐；通知服务慢了，可以先把任务放进队列；报表服务失败，不应该影响用户下单。微服务的目标之一，就是让一个地方出问题，不要把全站拖垮。

第五个优点是团队更容易分工。

服务边界清楚以后，一个团队可以负责一个服务的完整生命周期：开发、测试、上线、监控、故障处理。这比很多团队一起改同一个大项目更容易管理。

## 微服务的缺点

微服务的缺点也很明显。它不是免费的架构升级。

第一个缺点是复杂度上升。

单体里函数调用就是函数调用，微服务里变成网络请求。网络请求可能失败、超时、重复、延迟变高。以前一行代码能拿到结果，现在要考虑下游服务有没有挂、请求要不要重试、超时多久合适。

第二个缺点是数据一致性更难。

单体里一个数据库事务可以解决很多问题。微服务里，订单、库存、支付可能都有自己的数据库。下单时既要创建订单，又要扣库存，还要创建支付单，这就不能简单靠一个事务完成。

这时候常见做法是最终一致：订单先创建，再发消息给库存服务预占库存，支付成功后再推进订单状态。如果某一步失败，还要有补偿和重试。

第三个缺点是排查问题更难。

一个请求可能经过网关、订单、库存、支付、消息队列、通知服务。用户说“下单失败”，你要知道到底是哪个环节失败。没有日志、指标和链路追踪，就很难查。

第四个缺点是部署和运维成本更高。

服务多了以后，配置、密钥、数据库迁移、服务发现、负载均衡、监控告警、灰度发布都要管理。小团队如果没有这些能力，微服务会很累。

第五个缺点是容易变成分布式单体。

这是最常见的问题。服务拆开了，但共享数据库、互相强依赖、发布还要排队。这样既没有单体简单，也没有微服务灵活。

## 为什么用 Go 写微服务

Go 适合写微服务，主要是因为它简单、稳定、部署方便。

Go 编译后通常就是一个二进制文件，放进服务器或容器里就能跑。微服务数量一多，部署会变复杂，Go 的部署方式能少很多负担。

Go 的并发模型也适合后端服务。goroutine 很轻，适合处理大量网络请求。API 服务、RPC 服务、网关、任务消费者，这些都很适合用 Go 写。

Go 的标准库也够用。`net/http` 可以写 HTTP 服务，`context` 可以控制超时和取消，`testing` 可以写测试，`pprof` 可以做性能分析。特别是 `context`，微服务里非常重要，因为一次请求可能经过多个服务，超时和取消要能传下去。

另外 Go 的代码风格比较克制，不太鼓励写特别复杂的抽象。微服务本身已经有分布式复杂度，如果代码再绕来绕去，维护会更难。

但 Go 也不会自动让项目变清楚。Go 项目如果没有规范，也会写成一堆 handler、logic、model 混在一起。语言简单不等于架构简单，服务边界、错误处理、日志规范还是要认真设计。

## Go 微服务框架

Go 生态里有不少微服务框架，它们不是谁完全替代谁，而是侧重点不同。

### go-zero

go-zero 更偏“快速落地”。它提供 API 服务、RPC 服务、代码生成、配置、服务发现、负载均衡、链路追踪等能力。

如果你想快速搭一个比较规范的 Go 微服务项目，go-zero 是很直接的选择。它会帮你把很多基础结构先搭起来，不用从零开始设计 API、RPC、配置和服务治理。

它还有一个很有用的思路：API 层负责对外请求，RPC 层负责内部服务能力。这样外部入口和内部服务会比较清楚。

### Kratos

Kratos 更偏云原生服务框架。它重视 Protobuf、HTTP/gRPC、Middleware、日志、指标、链路追踪等能力。

如果团队比较重视接口契约，希望 HTTP 和 gRPC 都用统一方式组织，Kratos 是一个值得看的选择。它适合对工程规范要求比较高的项目。

### Kitex

Kitex 更偏高性能 RPC。它适合内部服务之间高频调用，尤其是对性能、扩展性、服务治理要求比较高的场景。

如果你的重点是服务间 RPC 通信，Kitex 很合适。但它不是完整业务脚手架，通常需要和其他组件搭配。

### Go kit

Go kit 更像工具包，而不是强约束框架。它给你一些微服务常用的组件和设计思路，但不会强迫你按某一种目录结构写。

如果团队已经有自己的工程规范，只想要更灵活的工具组合，可以看 Go kit。对初学者来说，它没有 go-zero 那么直接。

### 怎么选

想快速开始，可以先看 go-zero。

想要更完整的云原生服务骨架，可以看 Kratos。

重点是高性能 RPC，可以看 Kitex。

想要灵活组合工具，可以看 Go kit。

不过框架只是工具。服务边界没想清楚，换什么框架都救不了。

## 开发微服务要注意什么

第一，服务边界要清楚。

拆服务前先问：这个服务负责什么？不负责什么？它拥有哪部分数据？它失败会影响谁？

第二，不要共享数据库。

多个服务直接读写同一张表，是微服务里很危险的做法。短期可能方便，长期会让服务之间绑死。更好的做法是服务拥有自己的数据，其他服务通过接口或事件获取。

第三，接口要稳定。

服务之间通过接口协作，接口一变，上下游都可能受影响。字段含义、错误码、版本兼容都要谨慎。

第四，每次远程调用都要有超时。

不要让请求无限等待。下游服务慢了，上游也会被拖住，最后可能整条链路都慢。

第五，重试要小心。

重试只能用于临时错误，而且接口要幂等。否则可能造成重复下单、重复扣库存、重复发消息。

第六，日志和链路追踪要早点做。

微服务不是不能排查，而是要有工具。每个请求最好有 trace id，日志里要能看到它经过了哪些服务、在哪一步失败。

第七，能异步的就不要强行同步。

通知、报表、日志分析、长任务都适合异步。用户不一定要等所有事情都做完才拿到响应。

## 微服务功能怎么实现

一个微服务系统不只是几个服务，还需要一套配套能力。

### API 网关

网关是系统的入口。用户请求先到网关，网关再转发给后面的服务。

网关可以做鉴权、路由、限流、统一错误处理。这样客户端不需要知道后面有多少服务，只需要访问统一入口。

### 服务发现与负载均衡

服务发现解决“去哪找服务”，负载均衡解决“找到多个服务实例后调用哪一个”。

比如订单服务要调用库存服务，不应该写死某台机器的 IP，而应该通过 `inventory-service` 这个名字找到可用实例。

如果库存服务有 3 个实例，请求应该分散到不同实例上。最简单的是轮询，也可以按权重、最少连接、延迟优先等策略分配。

负载均衡不只是平均分流，也能提高可用性。如果某个实例不健康，请求就应该避开它。

### 服务间通信

服务之间常见三种通信方式：HTTP、RPC、消息队列。

HTTP 调试方便，适合对外接口和简单调用。RPC 契约更强，适合内部高频调用。消息队列适合异步任务、削峰和解耦。

不要为了“像微服务”而强行 RPC。通信方式应该根据业务来选。

### 服务治理

服务治理就是让服务在线上稳定运行的一组能力。

健康检查用来判断实例还能不能处理请求。如果实例不健康，就从可用列表里摘掉。

超时用来避免请求一直等待。

重试用来处理短暂抖动，但必须控制次数，并且接口要幂等。

限流用来保护服务不被打爆。

熔断用来保护调用方。如果下游一直失败，调用方先停止调用，避免把问题放大。

降级用来保住核心流程。比如推荐服务失败时返回热门内容，通知失败时稍后补发。

### 数据一致性

微服务里最难的通常是数据。

单体里一个事务能解决很多事。微服务里每个服务有自己的数据，就需要用事件、消息、补偿、幂等来保证最终一致。

比如下单流程可以是：订单服务创建订单，发送消息；库存服务消费消息预占库存；支付服务创建支付单；支付成功后再更新订单状态。

每一步都要考虑失败和重复。

### 配置管理

服务多了以后，配置会变多。数据库地址、Redis 地址、队列 Topic、限流参数、开关配置、密钥，都不能随便写死在代码里。

小项目可以先用配置文件和环境变量，规模大了再引入配置中心。

### 日志、指标和链路追踪

日志告诉你发生了什么。

指标告诉你系统是否健康，比如 QPS、错误率、延迟、CPU、内存。

链路追踪告诉你一个请求经过了哪些服务。请求跨服务时，trace id 非常重要。

### 部署、灰度和回滚

微服务强调独立部署，但独立部署也意味着发布次数更多。

上线前要有自动化测试、镜像构建、灰度发布和回滚方案。发坏了能及时发现，也能快速退回去。

## 一个更稳妥的落地路线

如果从零开始做 Go 后端，我不建议一开始就拆很多服务。

更稳妥的做法是先写模块化单体。先把用户、订单、支付、库存这些模块边界写清楚。代码在一个项目里，但模块之间不要乱依赖。

然后补工程基础设施：日志、配置、测试、数据库 migration、CI/CD、监控告警。

等某个模块确实有独立发布、独立扩容、独立维护的需求，再把它拆成服务。

第一次拆服务不要贪多。挑一个边界清楚的模块，跑通服务发现、接口调用、日志追踪、部署发布和回滚。

拆完还要观察。如果服务拆出来以后调用链变长、排查更麻烦、发布更痛苦，那说明边界可能没拆对。

微服务不是一条只能往前走的路。拆错了也可以合并，架构应该服务于项目，而不是项目服务于架构。

## 总结

微服务首先是一种拆分思想，其次才是框架和组件。

它能带来独立上线、职责清晰、单独扩容、故障隔离这些好处，但也会带来远程调用、数据一致性、排查、部署治理这些麻烦。

Go 适合写微服务，是因为它简单、稳定、好部署、并发能力强。go-zero、Kratos、Kitex、Go kit 都可以用来做 Go 微服务，但它们解决的问题不完全一样。

真正决定微服务质量的，不是用了多少组件，而是服务边界是否清楚、数据归属是否明确、接口是否稳定、出了问题能不能看见、发布失败能不能回滚。

微服务不是为了显得架构高级，而是为了让复杂系统还能被清楚地维护。
