> RAG 不是“向量数据库 + 大模型”，而是一套持续给模型供应可靠上下文的工程系统。

很多 RAG 项目一开始都很顺：文档切块、embedding、top-k 召回、拼进 prompt，大模型就能回答。这个 demo 版本很容易做出来，但真正上线后会遇到一堆问题：召回不到、召回太乱、答案不引用证据、权限泄漏、旧文档覆盖新文档、表格和代码块失真、用户问复杂问题时答非所问。

所以我现在更倾向于把 RAG 理解成“上下文工程”。检索只是其中一环，真正决定效果的是：知识如何治理，问题如何改写，证据如何召回和重排，上下文如何组织，模型如何被约束，以及结果如何评估。

## RAG 是什么

RAG 是 Retrieval-Augmented Generation，也就是检索增强生成。它的核心思想是：模型本身负责理解、推理和表达，外部知识库负责提供可更新、可追溯、可控制的事实依据。

传统大模型主要依赖参数记忆，但参数记忆有几个问题：训练后不容易更新，无法天然知道私有资料，遇到不确定内容时可能编造，而且很难说明答案来自哪里。RAG 通过外部检索把相关资料放进上下文，让模型基于这些资料回答。

> RAG 的目标不是让模型“凭空知道更多”，而是让模型在回答时拿到更可靠、更及时、更可追溯的上下文。

但这也说明了 RAG 的边界：它不是推理引擎，不是数据库，不是权限系统，也不是业务流程系统。它只是让模型获得外部证据的一种方式。

## 一个完整 RAG 的流程

完整 RAG 可以分成两个阶段：离线阶段处理知识，在线阶段处理问题。

<figure class="rag-flow">
  <figcaption>RAG 的基本链路</figcaption>
  <ol>
    <li><span>1</span><strong>资料收集</strong><em>文档、网页、数据库、代码、业务知识</em></li>
    <li><span>2</span><strong>清洗治理</strong><em>去噪、去重、权限、版本、时效性</em></li>
    <li><span>3</span><strong>切块建模</strong><em>保留标题层级、表格、代码和元数据</em></li>
    <li><span>4</span><strong>建立索引</strong><em>向量索引、关键词索引、元数据索引</em></li>
    <li><span>5</span><strong>查询处理</strong><em>改写、拆解、多查询、意图识别</em></li>
    <li><span>6</span><strong>召回重排</strong><em>hybrid search、rerank、过滤、去重</em></li>
    <li><span>7</span><strong>上下文拼装</strong><em>排序、压缩、引用、预算控制</em></li>
    <li><span>8</span><strong>生成校验</strong><em>基于证据回答、拒答、引用检查</em></li>
  </ol>
</figure>

很多失败的 RAG 都是只做了切块、索引、召回、生成，而且每一步都做得很粗糙。真正要做好 RAG，必须把整个链路当成一个系统来设计。

## RAG 开发过程中要注意的细节

### 1. 数据不是直接丢进去就行

RAG 的质量首先取决于知识库质量。文档需要清洗、去重、分层、标记来源和版本。尤其是企业或项目资料，必须处理权限、时效性和冲突信息。否则系统可能召回旧版本文档，或者把用户无权访问的资料放进上下文。

- 给每个 chunk 保存来源、标题路径、更新时间、作者、权限标签。
- 区分正式文档、草稿、历史版本、FAQ、日志、代码和表格。
- 删除导航栏、页脚、重复免责声明、广告等噪声文本。
- 保留文档结构，不要把标题、表格、代码块和正文混成一坨。

### 2. chunking 是 RAG 的第一道工程难点

切块不是简单按 500 token 一刀切。小块召回精准，但可能丢上下文；大块上下文完整，但会带来噪声。实际开发里更常用的是按标题层级切块、语义切块、滑动窗口、parent-child chunk。

一个常见实践是：用小 chunk 做召回，用父级段落或章节做上下文。这样可以让检索保持精准，同时让模型看到更完整的语义。

| 切块方式 | 优点 | 风险 |
| --- | --- | --- |
| 固定长度切块 | 实现简单 | 容易切断语义、标题丢失 |
| 按标题切块 | 结构清晰 | 标题质量差时效果不稳 |
| 语义切块 | 更符合自然段落 | 成本更高，需要额外模型或规则 |
| parent-child chunk | 兼顾精准和上下文 | 索引和拼装逻辑更复杂 |

### 3. embedding 不是随便选一个就完了

embedding 模型会影响召回上限。中文、代码、法律、医学、业务术语，对 embedding 的要求都不一样。更换 embedding 模型后通常需要重新建索引，也要重新跑评估集，而不是只看几个样例问题。

- 中文资料优先测试中文语义召回能力。
- 代码场景要测试函数名、错误码、类名、路径等精确检索。
- 长文档要测试跨章节问题，而不是只测单段问答。
- embedding 相似不等于答案相关，因此后面通常还需要 rerank。

### 4. 不要只依赖向量检索

纯向量检索擅长语义相似，但不一定适合精确匹配。比如版本号、专有名词、API 名、错误码、配置项，关键词检索往往更可靠。因此生产 RAG 经常使用 hybrid search：向量检索 + BM25 + 元数据过滤。

如果用户问“Go 1.22 的某个改动”，系统应该优先过滤版本和时间；如果用户问“某个报错怎么解决”，关键词命中可能比语义相似更重要。

### 5. query 需要处理

用户的问题经常不适合直接检索。比如“这个怎么部署”“为什么报错”“和上面那个有什么区别”，这些 query 缺少上下文。RAG 系统应该根据会话、页面上下文或用户意图，把问题改写成更适合检索的形式。

常见 query 策略包括：

- query rewrite：把口语问题改成完整问题。
- multi-query：生成多个不同表达方式进行召回。
- query decomposition：把复杂问题拆成多个子问题。
- HyDE：先生成一段假想答案或文档，再用它去检索。
- 意图识别：判断是问答、搜索、对比、总结，还是需要调用工具。

### 6. rerank 往往比继续调 embedding 更有用

召回阶段追求“不要漏”，重排阶段追求“排得准”。top-k 召回出来的内容可能只是语义相似，不一定能回答问题。reranker 可以把真正相关的证据排到前面，并减少无关 chunk 进入上下文。

如果系统经常出现“召回结果里有答案，但模型没用到”，可以先检查排序。相关证据排在上下文中间或很靠后时，模型更容易忽略。长上下文研究里也观察到模型对上下文中间位置的信息利用并不稳定，这就是常说的 lost in the middle 问题。

### 7. context packing 是独立问题

检索结果不能直接塞进 prompt。需要做去重、合并、排序、压缩和引用标注。上下文太少会缺证据，太多会稀释重点，甚至引入冲突信息。

- 把最关键证据放在靠前位置。
- 同一文档相邻 chunk 可以合并。
- 不同版本内容冲突时，按时间和权威等级处理。
- 保留来源编号，方便模型引用。
- 不要让检索内容覆盖系统指令，检索文本应被视为不可信输入。

### 8. 生成阶段要允许拒答

一个 RAG 系统如果永远回答，就一定会编。应该明确告诉模型：证据不足时直接说明无法判断。对高风险场景，还需要做二次校验，比如检查答案中的关键 claim 是否能在上下文找到支持。

### 9. 评估不能只看“回答像不像”

RAG 要分层评估。检索错了，生成再好也没用；检索对了，生成可能仍然幻觉。常见指标包括：

| 层级 | 指标 | 要回答的问题 |
| --- | --- | --- |
| 检索 | Recall@k、MRR、NDCG、context precision | 相关证据有没有被找出来，排序是否合理 |
| 生成 | faithfulness、answer relevance、citation accuracy | 回答是否基于证据，是否答到点上 |
| 系统 | 延迟、成本、拒答率、人工满意度 | 是否能稳定服务真实用户 |

RAGAS 和 ARES 这类工作都把 RAG 评估拆成了上下文相关性、答案相关性、忠实性等维度。实际项目里，最重要的是先做一套小而稳定的评测集，覆盖真实用户问题，而不是只跑几个漂亮样例。

## RAG 常见问题与排查方向

| 问题 | 常见原因 | 处理方向 |
| --- | --- | --- |
| 问得到资料里有，但系统答不出来 | 召回失败、query 太口语、chunk 切坏 | query rewrite、hybrid search、调整 chunk、增加 metadata filter |
| 召回结果很多但答案不准 | 噪声太多、rerank 缺失、上下文太乱 | rerank、MMR 去重、上下文压缩、提高引用约束 |
| 模型编造引用 | 没有强制来源编号，生成阶段约束弱 | 给 chunk 编号，要求每个关键结论绑定引用，做 citation check |
| 旧答案覆盖新答案 | 没有版本和时间过滤 | 加入更新时间、文档状态、权威等级，召回时过滤旧版本 |
| 权限泄漏 | 先检索后过滤，或没有 ACL metadata | 检索前做权限过滤，索引层保存用户/角色/组织权限 |
| 复杂问题答得浅 | 问题需要多跳检索或跨文档综合 | query decomposition、agentic retrieval、GraphRAG |
| 表格、代码、公式效果差 | 解析格式丢失，chunk 不适合结构化内容 | 表格转 Markdown/JSON，代码单独索引，保留字段和标题 |
| 延迟太高 | 多路召回、rerank、长上下文成本高 | 缓存、分层召回、减少 top-k、异步预取、轻量 reranker |
| 被 prompt injection 影响 | 检索文本中包含恶意指令 | 把检索内容标记为资料而非指令，过滤可疑内容，系统指令优先 |

## RAG 的各种进阶策略

### Naive RAG

最基础的方式：切块、embedding、top-k、生成。适合 demo 或资料很简单的场景，但上线后通常会遇到召回、噪声、引用和评估问题。

### Hybrid RAG

把向量检索、关键词检索和元数据过滤结合起来。它通常比纯向量检索更稳，尤其适合技术文档、代码、产品说明、FAQ 等有大量专有词的场景。

### Rerank RAG

先召回较多候选，再用 reranker 精排。适合“召回里有答案，但模型经常没用对”的情况。

### Corrective RAG

CRAG 的思路是先判断检索结果质量，如果质量不足，就触发纠正动作，比如重新检索、改写 query、使用外部搜索。它适合资料质量不稳定或 query 分布很杂的场景。

### Self-RAG

Self-RAG 强调让模型学会判断什么时候需要检索、检索结果是否有用、生成内容是否可靠。它的核心不是每次都检索，而是让检索变成可控动作。

### Agentic RAG

Agentic RAG 会把检索放进一个多步决策过程：先理解任务，再决定查什么，必要时多次检索、调用工具、检查结果、继续追问。它适合复杂问题，但成本和不可控性也更高。

### GraphRAG

GraphRAG 会把资料中的实体、关系和社区结构抽出来，用图结构帮助回答全局性、关系型、跨文档问题。比如“这个项目里哪些模块互相影响”“一个组织的关键主题是什么”，这种问题单靠 top-k chunk 很难答好。

### Long-context RAG

长上下文模型让我们可以放入更多资料，但它不是 RAG 的替代品。长上下文解决的是“能放多少”，RAG 解决的是“该放什么”。如果资料不筛选，长上下文只会把噪声一起放进去。

## 目前比 RAG 更合适的策略有哪些

严格说，没有一个方案天然“比 RAG 更好”。更准确的说法是：很多场景不应该用 RAG 解决，或者不应该只用 RAG。

| 场景 | 更合适的方案 | 原因 |
| --- | --- | --- |
| 查实时库存、订单、价格 | 数据库查询 / API 调用 | 需要精确、实时、可执行，RAG 容易把历史文本当事实 |
| 需要执行操作 | 工具调用 / Agent 工作流 | RAG 只能读资料，不能可靠完成业务动作 |
| 资料很少但很长 | 长上下文直接阅读 | 没必要建复杂索引，但仍要注意上下文位置和噪声 |
| 固定格式输出、风格迁移 | 微调 / few-shot 模板 | 问题不在知识检索，而在输出行为稳定性 |
| 强关系、多跳、全局总结 | GraphRAG / 知识图谱 | 需要关系结构，而不是孤立 chunk |
| 用户只是想搜索资料 | 传统搜索 + 摘要 | 不一定需要生成完整答案，搜索结果更透明 |
| 高风险决策 | 规则系统 + 人审 + 检索辅助 | 不能把最终判断完全交给生成模型 |

所以现在更好的方向不是“抛弃 RAG”，而是把 RAG 放到合适的位置：它负责提供证据，工具调用负责获取实时数据和执行动作，Agent 负责多步规划，长上下文负责阅读整份材料，微调负责稳定输出风格。

## RAG 上线前检查清单

- 有没有一套真实问题评测集，而不是只靠人工随便问？
- 每个 chunk 是否保存来源、标题路径、更新时间、权限和版本？
- 是否同时测试了向量检索、关键词检索和 hybrid search？
- 是否有 rerank，或者至少有相关性过滤？
- 是否允许证据不足时拒答？
- 回答中的关键结论是否能追溯到引用来源？
- 是否处理旧文档、新文档、草稿、正式文档之间的优先级？
- 是否防止未授权文档进入检索结果？
- 是否监控召回失败、幻觉、延迟、成本和用户反馈？
- 是否确认这个问题真的适合 RAG，而不是更适合 SQL、API、Agent 或长上下文？

## 总结

RAG 没有过时，过时的是 naive RAG。一个可靠的 RAG 系统应该能回答三个问题：知识是否被正确组织，证据是否被稳定召回，模型是否真的基于证据回答。

如果这三个问题没有解决，再大的模型、再好的向量数据库，也很难让 RAG 稳定可靠。反过来，如果这些环节做扎实，RAG 仍然是构建大模型应用时非常重要的基础设施。
